# OpenCompany Agent Principles

- 将 OpenCompany 构建为面向项目目录的、本地优先、模块化的 multi-agent 框架。
- 遵循 bitter lesson 和第一性原理：优先选择简单、可扩展的基础原语与直接设计，而不是沉重的人为编排。
- 开发与验证可使用名为 `OpenCompany` 的 Conda 环境，或等价的 `uv` 虚拟环境（Python 3.12 + dev 依赖）。
- 让根协调 agent 只做组织者，不做执行者：它可以查看上下文、规划、拆解任务、跟踪进展，并决定何时停止。
- 让 worker agent 执行实际任务，通过显式工具协作，并在结束时产出可供协调者消费的简洁总结。
- 每个 agent 都必须有稳定唯一 ID，并记录父子谱系与生命周期状态。
- 支持两种 session 工作区模式：`staged` 保留隔离可写工作区与显式 apply/undo，`direct` 允许在所选 sandbox backend 策略下直接写入目标项目目录（`anthropic` 为受约束，`none` 为无约束）。`direct` 下工作目录可以是本地目录，也可以是远程 SSH Linux 目录；`staged` 必须在 UI 配置层和后端校验层都拒绝远程工作目录。所选模式必须在整个 session 生命周期内保持不变。
- 将编排限制保持为显式且可配置，包括子 agent 扇出数、活跃 agent 数、步骤预算和根循环上限。
- 当前运行策略偏离：当循环或预算触达上限时，以软提醒推动总结并优先回交用户，不做硬性强制终止。
- 保持 agent 工具最小且可组合：项目查看、sandbox 终端、创建 subagent、终止 subagent、agent steer、查询状态。
- 保持编排流程模块化且易于替换，使未来调整协作流程时不需要整体重写。
- 保持 LLM 后端可插拔；当前首个支持的后端是带流式能力的 OpenRouter。
- 所有 prompt 必须集中放在一个目录中，并可版本化管理。
- 提供双本地图形界面（终端 + Web），支持切换目录、设置、输入任务、查看 agent 活动与中断执行。
- 所有关键动作、工具调用和 agent 间通信都要结构化记录，便于回放、调试与数据收集。
- 优先使用明确的限制、checkpoint 和可恢复性，而不是隐藏的长生命周期状态。
- 结合模块化测试与端到端系统测试，以避免回归并减少重复掉入同类失败陷阱。
- 全量测试耗时通常较长（约数分钟）；在判断卡住前请耐心等待完成。
- 在锁定重要技术路线前先完成充分调研；如果不确定性会实质影响设计，需要先与用户讨论再定。
- 随着系统演进，持续更新 `docs/`、`README.md` 和 `AGENTS.md`。
- 将 `docs/README.md` 作为文档总索引；文档结构变化时必须同步更新该索引。
- 在 `docs/modules/` 维护模块级文档；行为改动时需同次变更更新对应模块文档。
- 英文与中文镜像文档需保持同结构（章节与顺序一致）。
- prompt、文档与 README 以英文为主，并维护命名为 `*_cn.md` 的同步中文版。
- 界面语言默认跟随本地语言；如果既不是中文也不是英文，则回退到英文。
